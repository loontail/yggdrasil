# Skin & cape upload

Four methods on `YggdrasilClient` cover the user-facing texture lifecycle. They
all carry the user's `accessToken` in the `Authorization: Bearer …` header;
the server identifies the owner from that token.

| Method | Purpose |
|---|---|
| `client.uploadSkin({ accessToken, file, variant? })` | Upload or replace the caller's skin. |
| `client.uploadCape({ accessToken, file })` | Upload or replace the caller's cape. |
| `client.deleteSkin({ accessToken })` | Remove the caller's skin. |
| `client.deleteCape({ accessToken })` | Remove the caller's cape. |

## Uploading

```ts
import { SkinVariants } from '@loontail/yggdrasil-client';
import { readFile } from 'node:fs/promises';

const png = await readFile('./steve.png');

await client.uploadSkin({
  accessToken,
  file: png,
  variant: SkinVariants.CLASSIC,
});
```

The `file` argument accepts either a `Uint8Array` or an `ArrayBuffer`. In the
browser, pass `await file.arrayBuffer()` from a `<input type="file">`.

The request hits `PUT /textures/skin` with a `multipart/form-data` body:

```
--boundary
Content-Disposition: form-data; name="file"; filename="asset.png"
Content-Type: image/png

<PNG bytes>
--boundary
Content-Disposition: form-data; name="variant"

CLASSIC
--boundary--
```

The boundary is auto-generated by the runtime's `FormData` implementation.

## Variants

Skins carry a "model" — `CLASSIC` (Steve, 4-pixel arms) or `SLIM` (Alex,
3-pixel arms). Capes do not.

```ts
import { SkinVariants } from '@loontail/yggdrasil-client';

SkinVariants.CLASSIC; // 'CLASSIC'
SkinVariants.SLIM;    // 'SLIM'
```

If you don't pass `variant`, the server defaults to `CLASSIC`. The vanilla
client renders accordingly — passing `SLIM` for a skin drawn with 4-pixel arms
will produce visual gaps; passing `CLASSIC` for a slim skin will overlap the
arm texture onto the body. There's no way to detect the right variant from the
PNG pixels deterministically; the *uploader* knows which model the skin was
drawn for.

For legacy skin rows where the variant wasn't recorded (e.g. migrated from a
pre-plugin skin-registry), the plugin falls back to
`detectMojangSkinVariant(buffer)` from `@loontail/minecraft-kit`, which uses
Mojang's own transparency heuristics.

## Client-side PNG validation

Before the network request is built, the client runs:

```ts
import { assertPngBuffer } from '@loontail/yggdrasil-core';

assertPngBuffer(file, 'skin'); // or 'cape'
```

This is a byte-level check (not a `Content-Type` check):

- Length ≥ 24 bytes.
- First 8 bytes match the PNG signature.
- The first chunk type is `IHDR`.
- Dimensions match the kind's allowed list:
  - Skins: `64×64` or `64×32`.
  - Capes: `64×32`.

A failure throws `YggdrasilCoreError(invalid_png)` synchronously — *before*
the request hits the wire. The error's `context.reason` is a human-readable
string:

```
file too small to be a PNG (need at least 24 bytes)
file is not a PNG (header 89504E47 0D0A1A0A …)
first PNG chunk is "sBIT", expected "IHDR"
skin dimensions 128x128 are not supported (expected 64x64 or 64x32)
```

The server runs the same validator on its side. The client-side check is
purely a friendliness layer — broken uploads fail without a round-trip and
without consuming the user's bandwidth.

## Server-side validation

Even though the client validates, the server can't trust that — it runs
`assertPngBuffer` again. It also enforces:

- **Size cap: 256 KB.** Files larger than this are rejected with HTTP 400 and
  the Yggdrasil error envelope `IllegalArgumentException`.
- **Token validity.** The `yggdrasil-token-auth` policy looks the
  `accessToken` up in `yggdrasil_tokens` and refuses if it's missing or
  expired (HTTP 403, `ForbiddenOperationException`).

## Deletion

```ts
await client.deleteSkin({ accessToken });
await client.deleteCape({ accessToken });
```

Idempotent — calling delete twice doesn't fail (the server returns 204 either
way). The on-disk PNG and the DB row are both removed.

## Reading another player's textures

The protocol-compliant path:

```ts
const profile = await client.profile(uuid, { signed: true });
const texturesProp = profile.properties?.find((p) => p.name === 'textures');
if (!texturesProp) return null;

const payload = decodeTexturesPayloadBase64(texturesProp.value);
payload.textures.SKIN?.url;            // string
payload.textures.SKIN?.metadata?.model; // 'slim' | undefined
payload.textures.CAPE?.url;            // string
```

The non-protocol convenience path:

```ts
const textures = await client.getTextures(uuid);
// { skin: { url, variant } | null, cape: { url } | null }
```

The convenience endpoint is simpler but doesn't carry a signature. Use it for
launcher UI; use the signed profile for anything security-sensitive.

## Sizing PNGs

The texture format Mojang expects is `64×64` (modern) or `64×32` (legacy "short
skin"). HD skins (`128×128`, `256×256`) require server-side opt-in in
authlib-injector via `meta.feature.skin_size`, which this plugin does *not* set.
Sticking to `64×64` keeps the server compatible with every vanilla client out
there.

Capes are always `64×32`. The visible region is the inner `22×17` pixels of
the top-left quadrant; the rest is alpha. Outside services like
[capes.dev](https://capes.dev) are good references.

## Multipart boundary issues

If you see a `400 IllegalArgumentException: missing file field` from a custom
HTTP layer, the most common cause is overriding `Content-Type` manually on the
request. Don't — let `fetch` set it from the `FormData` instance so the
boundary matches the body. `YggdrasilClient` does this correctly out of the
box; if you wrote a custom `fetch` adapter, make sure it doesn't strip or
rewrite the multipart `Content-Type`.
